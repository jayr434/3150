Refresher: Raw, Unique, Shared, and Weak Pointers
IMPORTANT: This section is just to refresh some of these ideas. You do NOT need to code
anything at all for the refresher section.
Before starting this homework, let us briefly review the four pointer types that will show
up throughout the assignment:
• raw pointers
• std::unique ptr
• std::shared ptr
• std::weak ptr
We are going to use these across the phases of this homework to build increasingly safer
linked list implementations.

Refresher Step 1: Raw pointers
A raw pointer is just a memory address. It does not automatically clean up memory for you.
#include <iostream>
int main() {
int* x = new int(42);

1

std::cout << *x << "\n";
delete x;
x = nullptr;
return 0;
}
This is very lightweight and very fast, but it is also easy to make mistakes:
• forgetting to call delete
• calling delete twice
• using a pointer after the memory has already been freed
So raw pointers give you a lot of control, but also a lot of responsibility.

Refresher Step 2: std::unique ptr
A unique ptr owns one object exclusively. When the unique ptr goes away, it automatically
deletes the object it owns.
#include <iostream>
#include <memory>
int main() {
std::unique_ptr<int> x = std::make_unique<int>(42);
std::cout << *x << "\n";
return 0;
}
Here, we do not need to manually call delete. That is the big benefit.
A unique ptr cannot be copied, because ownership must stay unique. It can only be
moved.
std::unique_ptr<int> a = std::make_unique<int>(5);
// std::unique_ptr<int> b = a;
// not allowed
std::unique_ptr<int> b = std::move(a);
This is often the cleanest smart pointer when exactly one thing should own an object.
2

Refresher Step 3: std::shared ptr
A shared ptr allows multiple owners of the same object.
#include <iostream>
#include <memory>
int main() {
auto a = std::make_shared<int>(42);
auto b = a;
std::cout << *a << "\n";
std::cout << *b << "\n";
std::cout << a.use_count() << "\n";
return 0;
}
Here, both a and b point to the same integer. The object stays alive as long as at least
one shared ptr still owns it.
This is convenient, but it has overhead:
• reference counting must be maintained
• ownership can become less clear

Refresher Step 4: std::weak ptr
A weak ptr points to an object managed by shared ptr, but it does not own it.
#include <iostream>
#include <memory>
int main() {
auto a = std::make_shared<int>(42);
std::weak_ptr<int> w = a;
std::cout << a.use_count() << "\n";
if (auto locked = w.lock()) {
std::cout << *locked << "\n";
3

}
return 0;
}
The key idea is:
• shared ptr owns
• weak ptr observes
This becomes extremely important when there are cycles.
For example, if object A owns object B with a shared ptr, and object B also owns object
A with a shared ptr, then neither object can ever be freed. That creates a memory leak.
A weak ptr breaks that ownership cycle.

Phase 2 Rebuild the Singly Linked List with unique ptr

Now let us rebuild the singly linked list, but this time with unique ptr instead of raw
pointers.
The high-level data structure is the same:
• each node stores a value
• each node points to the next node
But now ownership is much clearer:
• each node uniquely owns the next node
• the list uniquely owns the head node
This is actually a very natural fit for a singly linked list.

Suggested structure
A good starting point would look like this:
#pragma once
#include <iostream>
#include <memory>
class LinkedList {
private:
struct Node {
int value;
std::unique_ptr<Node> next;
Node(int value) : value(value), next(nullptr) {}
};
std::unique_ptr<Node> head_;
Node* tail_;
public:
LinkedList();
~LinkedList() = default;
LinkedList(const LinkedList&) = delete;
LinkedList& operator=(const LinkedList&) = delete;
void push_back(int value);
void print(std::ostream& os) const;
};

Why is the tail pointer still raw?
You may notice something interesting here:
• head is a unique ptr<Node>
• tail is still just a raw pointer
That is okay.
9

The raw pointer here is not owning anything. It is just a convenient non-owning pointer
so that appending can be efficient.
Ownership still flows through the unique ptr chain:
• the list owns the head
• each node owns the next node
So when the head is destroyed, the entire chain will be destroyed automatically.

However, there is a hidden problem:

That means there is a reference cycle.
Even when the list object itself goes away, the nodes may still keep each other alive

If the cycle exists, you may notice that the node destructors do not all run as expected.
You can also compile with sanitizers to make memory problems easier to detect.
For example:
g++ -std=c++20 -Wall -Wextra -pedantic -fsanitize=address -g main.cpp -o main
And if your compiler supports leak sanitizer:
g++ -std=c++20 -Wall -Wextra -pedantic -fsanitize=address,leak -g main.cpp -o main
These flags are very useful during development and debugging.

Phase 4 Fix the Doubly Linked List with weak ptr

Now let us fix the ownership cycle.
We still want:
• forward links
• backward links
But we do not want both directions to own each other.
The standard solution is:
• make next a shared ptr
• make prev a weak ptr
That way:
• the forward chain owns the nodes
• the backward links simply observe

Suggested structure
Start from this:
#pragma once
#include <iostream>
#include <memory>
class DoublyLinkedList {
private:
struct Node {
int value;
std::shared_ptr<Node> next;
std::weak_ptr<Node> prev;

DoublyLinkedList& operator=(const DoublyLinkedList&) = delete;

Your tasks for this phase
Implement:
• push back
• print forward
• print backward
When moving backward through the list, remember that a weak ptr must be converted
temporarily into a shared ptr using lock().
For example:
auto previous = current->prev.lock();
If the object still exists, lock() gives you a usable shared ptr. If the object no longer
exists, it gives you an empty smart pointer.

Discussion: Performance Versus Clean Design

A few final thoughts on these exercises:
For a doubly linked list, raw pointers are generally more efficient than using shared ptr
and weak ptr. Smart pointers add overhead:
• shared ptr maintains reference counts
• weak ptr must coordinate with that ownership system
• locking a weak ptr is more work than just following a raw pointer
So if your goal is absolute performance, raw pointers will often be faster.
However, the smart pointer version is cleaner and safer. It makes ownership more explicit
and reduces the chance of memory management mistakes.
This is something you will need to balance as a C++ developer:
• faster code is often less clean
• cleaner code is often a little slower
For a small doubly linked list, this difference is not a problem at all. But for a course
project that contains a graph with 500,000 edges, that overhead matters considerably.
That does not mean you should start optimizing everything right away.
In fact, I strongly recommend that you build the entire course project first before optimizing anything. Get the code correct first. Get it working first. Then profile it. Then
optimize the parts that actually matter.
Premature optimization is one of the easiest ways to make a project harder to finish.
