## Part 2 Design Decisions

### AS struct (`as.hpp`)

```cpp
struct AS {
    uint32_t asn;
    std::vector<uint32_t> providers;  // ASNs only, not pointers
    std::vector<uint32_t> customers;
    std::vector<uint32_t> peers;
};
```

Stores ASNs rather than pointers. All lookups go through `ASGraph::ases` (an `unordered_map<uint32_t, AS>`). This avoids pointer invalidation on map rehash and is simpler to reason about.

### ASGraph (`as_graph.hpp` / `as_graph.cpp`)

- `unordered_map<uint32_t, AS> ases` — the graph. Reserved to 100k upfront.
- `addProviderCustomer(provider, customer)` — both `try_emplace` calls happen before any `operator[]` access to avoid iterator invalidation on rehash.
- `addPeer(asn1, asn2)` — bidirectional, same safety pattern.
- `fromStream(istream&)` / `fromFile(string&)` — split so tests don't need real files.
- **Cycle detection uses Kahn's algorithm** (iterative, not recursive DFS — avoids stack overflow on deep graphs). In-degree = number of customers. Leaves (no customers) process first; if not all ASes are processed, a cycle exists. **This same algorithm will give you propagation ranks for free** — the processing order is exactly rank 0 (leaves) → rank 1 → … → root. Save the order from Kahn's instead of throwing it away.

### CAIDA file format

```
# comment
provider_asn|customer_asn|-1|source    ← provider-customer
asn1|asn2|0|source                     ← peer-to-peer
```

Source column ignored. Any relationship value other than -1 and 0 is silently skipped.

### CLI (decided, not yet implemented in main.cpp beyond arg check)

```
./bgp_sim <topology_file> <anns.csv> <rov_asns.csv> <output.csv>
```

One topology file. Grader passes a pre-decompressed plain-text file. The test directories each contain two identical CAIDA files (`.10.15.txt` and `.10.16.txt`) — treat them as the same, just pass one.

### Cycle error behavior

```
exit(1)  +  message to stderr
```

Spec says "error out with the proper return code" without specifying a code. Non-zero exit is the Unix convention.

## Test Data — What Each Scenario Tests

All three test dirs share the same graph format: `CAIDAASGraphCollector_YYYY.MM.DD.txt` (same pipe-delimited CAIDA format, plain text, ~489k lines). Expected output is `ribs.csv`.

### `prefix/`
- AS 27 announces `1.2.0.0/16` (`rov_invalid=False`) — legitimate origin
- AS 25 announces `1.2.0.0/16` (`rov_invalid=True`) — prefix hijack
- ROV ASes: 174, 43, 68, 27
- Non-ROV ASes accept AS 25's hijack (shorter path wins). ROV ASes filter it and fall back to AS 27's legitimate path.

### `subprefix/`
- AS 27 announces `1.2.0.0/16` (`rov_invalid=False`)
- AS 25 announces `1.2.3.0/24` (`rov_invalid=True`) — more-specific subprefix hijack
- Same ROV ASes as prefix scenario
- These are **different prefixes** so they never conflict in the local RIB — each AS stores both independently. No longest-prefix-match logic needed; the RIB key is the exact prefix string.

### `many/`
- ~20 seed ASes each announcing 2 prefixes
- AS 25 hijacks AS 1's two prefixes (`10.0.0.0/24`, `10.0.1.0/24`) with `rov_invalid=True`
- Most seed ASes are also ROV ASes, so AS 25's hijack is filtered by them
- Scale: expected `ribs.csv` is ~3 million rows (~79k ASes × ~20 prefixes × 2 per prefix)

## Critical: Output Format

The expected output was generated by a Python reference implementation. The `as_path` column uses **Python tuple notation**:

```
asn,prefix,as_path
1,10.0.0.0/24,"(1,)"
1,10.3.0.0/24,"(1, 11537, 10886, 4)"
```

Rules:
- Single element: `(1,)` — trailing comma, no space before paren
- Multiple elements: `(1, 11537, 4)` — space after every comma, no trailing comma
- The entire as_path value is quoted in the CSV when it contains commas (standard CSV quoting)
- `compare_output.sh` does **exact string comparison** after sorting — any deviation in spacing or comma placement fails every row

Write a dedicated format function and test it in isolation before wiring it to the propagation output.

## Part 3 Implementation Guide

### What to build (in order)

**3.1 Announcement struct** — prefix (string), as_path (vector\<uint32_t\>), next_hop_asn (uint32_t), recv_relationship (enum: Origin > Customer > Peer > Provider).

**3.2 Policy / BGP / ROV classes**
- Abstract `Policy` base class
- `BGP` subclass: holds local RIB (`unordered_map<string, Announcement>`) and received queue (`unordered_map<string, vector<Announcement>>`)
- `ROV` subclass of `BGP`: drops announcements where `rov_invalid == true` on receive
- Each `AS` should hold a `unique_ptr<Policy>` (or raw pointer to a `Policy`)

**3.3 Graph flattening** — Reuse Kahn's from `hasCycle()`. Instead of discarding the processing order, store it as `vector<vector<uint32_t>>` where index = rank (0 = leaves, max = roots). Add a `flatten()` method to `ASGraph` that returns this structure (or stores it as a member).

**3.4 Seeding** — Parse `anns.csv` (header: `seed_asn,prefix,rov_invalid`). For each row, create an Announcement with relationship=Origin, as_path=[seed_asn], next_hop=seed_asn, store it in that AS's local RIB.

**3.5 Propagation** — Three phases, in order:
1. **Up** (rank 0 → max): each AS sends all local RIB entries to its providers' received queues. Then process received queues at rank 1, store (with conflict resolution), clear queue, send to providers. Repeat up to max rank.
2. **Across** (peers, one hop only): ALL ASes send first, THEN all ASes process and store. Do not interleave — this prevents multi-hop peer propagation.
3. **Down** (max rank → 0): reverse of up. Send from local RIB to customers' received queues, process, store, clear, repeat downward.

On each send: set `next_hop_asn` = sending AS's ASN.
On each store: prepend storing AS's ASN to `as_path`.

**3.6 Conflict resolution** — When an AS already has an announcement for a prefix and receives another:
1. Better relationship wins (Origin > Customer > Peer > Provider)
2. Tie: shorter as_path wins
3. Tie: lower next_hop_asn wins

**3.7 ROV** — Parse `rov_asns.csv` (no header, one ASN per line). Mark those ASes to use the `ROV` policy. ROV policy drops any announcement with `rov_invalid=true` before it can enter the received queue or local RIB.

**3.8 Output** — After propagation, iterate all ASes, iterate their local RIBs, write one row per (asn, prefix) pair. Format as_path using Python tuple notation (see above).

### Input file format notes

`anns.csv`:
```
seed_asn,prefix,rov_invalid    ← has header
1,10.0.0.0/24,False
25,10.0.0.0/24,True
```

`rov_asns.csv`:
```
174                            ← NO header, raw ASNs one per line
43
68
```

### Performance notes

- The `many/` scenario is the scale test: ~79k ASes, ~3M output rows
- The propagation loop is the hot path — keep inner loops lean
- `unordered_map` lookups in the inner propagation loop add up; consider whether the local RIB and received queue should use string keys or a hashed prefix type
- Spec allows up to 2 CPUs and 8GB RAM
